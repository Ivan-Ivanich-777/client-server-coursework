# Стратегия тестирования TaMP

## 1. Цели тестирования

- Убедиться в корректности реализации протокола (парсинг и формирование сообщений).
- Проверить бизнес-логику: авторизация и регистрация.
- Проверить работу базы данных: создание таблиц, CRUD-операции.
- Обеспечить регрессионную защиту при доработке сервера.

## 2. Виды тестирования

### 2.1 Unit-тесты (автоматические, Qt Test)

**Объект:** модуль `protocol` — функции `parseRequest()` и `makeXxx()`.  
**Инструмент:** Qt Test (`tst_protocol.cpp`).  
**Принцип:** каждая функция тестируется изолированно, без зависимостей на сеть или БД.

### 2.2 Интеграционные тесты (автоматические, Qt Test)

**Объект:** `ServerLogic` + `DbHandler` в связке.  
**Инструмент:** Qt Test (`tst_logic.cpp`), in-memory SQLite (`:memory:`).  
**Принцип:** тесты поднимают реальную БД в памяти, проверяют полный цикл обработки команды.

### 2.3 Ручное тестирование (системное)

**Инструмент:** PuTTY / Telnet (Raw-режим, порт 12345).  
**Принцип:** живое подключение к запущенному серверу (нативно или в Docker), отправка команд вручную и наблюдение ответов.

## 4. Запуск автоматических тестов

```bash
cd TaMP_Tests
cmake -B build -DCMAKE_PREFIX_PATH="C:/Qt/6.11.0/mingw_64"
cmake --build build
cd build
ctest --output-on-failure
```

Каждый тест-исполняемый файл также можно запустить напрямую:

```bash
./tst_protocol      # unit-тесты протокола
./tst_logic         # интеграционные тесты логики
```

## 5. Критерии приёмки

- Все автоматические тесты проходят без ошибок (`PASS`).
- При ручном тестировании каждая команда возвращает ожидаемый ответ согласно [таблице тест-кейсов](test_cases.md).
- Сервер в Docker принимает соединения и корректно обрабатывает команды через PuTTY.
